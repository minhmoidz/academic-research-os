# Evidence Ledger and Claim Tracing

## Overview

The evidence ledger is the central accountability mechanism of the Academic Research OS. It ensures that every claim written in the paper is traceable to a specific, documented piece of evidence — not to the author's recollection, not to training-data knowledge, and not to a paraphrase of a result that was never actually measured.

The ledger is the third link in a four-link chain that connects raw experimental data to published prose.

---

## 1. The Two-Hop Chain

```
results.tsv
    ↓
evidence_ledger.md
    ↓
claim-evidence-table.md
    ↓
paper.tex
```

Each link in the chain has a specific function:

**`results.tsv`** — Raw measurement log. Every experimental run appends one row: run ID, timestamp, configuration, metric name, metric value, status. This file is append-only. No values are ever edited after writing.

**`evidence_ledger.md`** — Curated evidence registry. Each approved result from `results.tsv` (and each relevant literature citation) receives a unique evidence ID and a quality assessment. The ledger records what was measured, under what conditions, and how confident we are in the measurement.

**`claim-evidence-table.md`** — Claim-to-evidence mapping. Each claim in the paper (a row in the table) is linked to one or more evidence IDs. A claim with no evidence ID is a floating claim and is forbidden after the Evidence Freeze.

**`paper.tex`** — The submitted manuscript. Each quantitative or comparative claim carries an inline comment (`% EVID-EXP-7`) linking to the specific evidence ID that supports it. This allows rapid verification during rebuttal.

The two-hop chain means: to challenge any claim in the paper, a reviewer can follow the chain from `paper.tex` to `claim-evidence-table.md` to `evidence_ledger.md` to `results.tsv` and find the exact run that produced the number.

---

## 2. Evidence ID Types

The OS uses four evidence ID types, each with a distinct prefix:

### EVID-EXP-N — Experimental Result

An original measurement produced by running the evaluation pipeline in this project.

Example: `EVID-EXP-7` — the result of running the proposed method on the ScientificQA test set with re-ranking enabled, producing recall@10 = 0.71 ± 0.02.

Use for: every quantitative result reported in the paper's main table, ablation table, and within-paper comparisons.

### EVID-LIT-N — Literature Citation

A factual claim sourced from a published external paper, verified by reading the paper directly (not from secondary sources or memory).

Example: `EVID-LIT-3` — the claim that DPR achieves recall@10 = 0.67 on ScientificQA, sourced from Table 2 of \cite{johnson2023}.

Use for: baseline numbers that were not re-run in this project, factual claims about prior methods, dataset statistics from the dataset's original paper.

### EVID-DATA-N — Dataset / Data Observation

A property of a dataset or corpus that is documented by inspecting the data directly (e.g., counting instances, measuring a distribution).

Example: `EVID-DATA-1` — the ScientificQA test split contains 1,200 queries with an average of 4.3 relevant documents per query (verified by running `python scripts/dataset_stats.py`).

Use for: dataset statistics, data quality observations, class balance claims, split size claims.

### EVID-REPO-N — Repository / Artifact Reference

A code artifact, configuration file, or reproducibility artifact that is the definitive reference for a methodological description.

Example: `EVID-REPO-2` — the re-ranking implementation is at `src/agent/reranker.py`, commit `a3f9d2c`.

Use for: implementation detail claims, hyperparameter claims, architecture descriptions where the code is the ground truth.

## 3. How to Create a Ledger Entry

Each entry in `evidence_ledger.md` follows this format:

```markdown
## EVID-EXP-7

- **Type:** Experimental result
- **Run ID:** run-009
- **Config:** configs/rerank_v4_final.yaml
- **Config diff:** results/run-009/config_diff.txt
- **Dataset:** ScientificQA test split (N=1,200 queries)
- **Evaluation command:** python eval/run_eval.py --config configs/rerank_v4_final.yaml
- **Metric:** recall@10 = 0.714 ± 0.019 (5-fold cross-validation)
- **Baseline reference:** EVID-EXP-1 (recall@10 = 0.612 ± 0.021)
- **Delta from baseline:** +0.102 (+16.7%)
- **Statistical test:** Paired t-test, p = 0.003 (N=5 folds)
- **Timestamp:** 2024-11-20T09:14:33Z
- **Git commit:** a3f9d2c
- **Status:** APPROVED
- **Supports claims:** C-3, C-5, C-8
```

**Required fields:** Type, Run ID (or citation key for LIT), Dataset, Metric, Timestamp, Status, and Supports claims.

**Optional but recommended:** Delta from baseline, statistical test, git commit.

**Status values:**
- `DRAFT` — auto-generated by the experiment loop; not yet reviewed
- `APPROVED` — reviewed by the researcher; may be used in the paper
- `REJECTED` — measurement was invalid (e.g., test set contamination discovered); must not be used
- `SUPERSEDED` — replaced by a later run with the same configuration; the later run's ID is noted

Only APPROVED entries may appear in `claim-evidence-table.md`.

## 4. How to Link Claims to Evidence in claim-evidence-table.md

`claim-evidence-table.md` is built during Stage 18 (Outline) and maintained through Stage 19 (Section Drafting). Each row corresponds to one claim in the paper:

```markdown
| Claim ID | Claim text (abbreviated) | Evidence IDs | Section | Status |
|----------|--------------------------|--------------|---------|--------|
| C-3 | "Our method achieves recall@10 of 0.714 on ScientificQA" | EVID-EXP-7 | §4.2 | SUPPORTED |
| C-4 | "This represents a 16.7% improvement over the DPR-Sci baseline" | EVID-EXP-7, EVID-LIT-3 | §4.2 | SUPPORTED |
| C-5 | "Re-ranking is the primary driver of improvement (ablation)" | EVID-EXP-9 | §4.3 | SUPPORTED |
| C-6 | "The method generalizes to the BioASQ dataset" | TODO_RESULT_NEEDED: generalization experiment not yet run | §4.4 | MISSING |
```

**Rules:**
- Every claim in `paper.tex` must have a row here
- Every row with status SUPPORTED must have at least one APPROVED evidence ID
- Rows with status MISSING must use `TODO_RESULT_NEEDED:` as the evidence value — never invent evidence
- After the Evidence Freeze, no new MISSING rows may be added without re-running Gate 3

---

## 5. The Evidence Freeze: What It Is, When It Triggers, What Gets Locked

The Evidence Freeze is Stage 17 of the workflow. It is initiated by running `/freeze-evidence` after Gate 3 returns Decision A or B.

**What it locks:**
- All rows in `evidence_ledger.md` (no new APPROVED entries may be added)
- All rows in `claim-evidence-table.md` (no new claims may be added without a revision cycle)
- The freeze timestamp in `project_state.md`

**What it does not lock:**
- `paper.tex` — writing continues after the freeze
- `results.tsv` — new runs may still be appended (but their evidence IDs cannot enter the ledger without re-running Gate 3)
- `references.bib` — citation additions for related work are allowed

**When it triggers:**
The freeze triggers immediately after Gate 3 Decision A or B is recorded in `adequacy_decision.md`. It cannot be triggered manually before Gate 3 — doing so would freeze an incomplete evidence set.

**Revision cycle:**
If a reviewer requests an additional experiment after submission, the project enters a revision cycle. The revision cycle opens a new evidence block (EVID-EXP-R1, EVID-EXP-R2, etc.) under a separate header `## Revision Evidence`. The revision block goes through a mini Gate 3 evaluation before being incorporated. The original freeze block is not modified.

## 6. What to Do When a Claim Has No Evidence

When drafting a section, if you want to make a claim that is not yet supported by an approved evidence entry, you must use a TODO marker — never invent or estimate the value.

The correct marker is:

```latex
% TODO_RESULT_NEEDED: [description of the experiment needed]
Our method achieves \textbf{XX.X\%} recall@10 on the BioASQ benchmark.
```

The `XX.X` placeholder signals to any reader (and to the OS's artifact scanner) that this is unconfirmed. Do not write a plausible-looking number as a placeholder. Do not write "approximately" or "roughly" followed by an estimated value.

Other TODO marker types for evidence-related gaps:

- `TODO_EVIDENCE_NEEDED:` — a claim exists in the outline but no experiment has been designed to support it
- `TODO_CITATION_NEEDED:` — a literature claim needs a citation that has not been verified
- `TODO_NOVELTY_CHECK_NEEDED:` — a novelty claim is pending Gate 1 clearance
- `TODO_VENUE_FIT_NEEDED:` — a claim about venue appropriateness is pending Gate 2

The OS scans for TODO markers before the Evidence Freeze. If any `TODO_RESULT_NEEDED:` markers exist in `paper.tex` at freeze time, the freeze fails and the missing experiments must be run or the claims must be removed.
